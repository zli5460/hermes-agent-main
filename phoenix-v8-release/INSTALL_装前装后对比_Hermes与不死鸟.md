# Hermes × 不死鸟（Phoenix）— 装前 vs 装后 · 详细对比

> **用途**：给学员、销售、自己人统一口径；章节可直接映射到 **信息图 / 幻灯片 / 画板**（每一小节 ≈ 一屏或一张卡）。  
> **前提**：「装不死鸟」指按交付包 **`install.sh` / `install.ps1`** 完整安装 Phoenix V8 并与 Hermes 融合后的状态。

---

## 1. 一句话

| 状态 | 一句话 |
|------|--------|
| **未装不死鸟** | Hermes Agent **用一套默认模型 + 一条线路 + 一把 Key** 托底；没有 Phoenix 的五档手动挡、专用记忆/自愈编排与 `phoenix_open` 配置层。 |
| **已装不死鸟** | 仍是 **同一个 Hermes**；在上面叠了 **插件 + `~/.hermes/phoenix`**：**日常照旧**，贵档仅在你输入 **`/深度` `/大神` `/真神` 并确认** 后切换；失败走兜底链，**执行完回到默认档**。 |

## 2. 总对比表（建议整表做一张「左右对照图」）

| # | 维度 | 未装不死鸟的 Hermes | 已装不死鸟的 Hermes × Phoenix |
|---|------|---------------------|--------------------------------|
| 1 | **产品边界** | 原生 Hermes 能力边界 | Hermes **不变**；Phoenix **扩展**路由与配置，**不抢 `hermes` 命令** |
| 2 | **默认聊天** | `config.yaml` 里 **`model.default`** 等决定的单一入口 | **同一入口**；日常对话 **不自动升档** |
| 3 | **高档能力** | 需用户自己 **`/model`** 或改配置换强模型（无统一「任务级」确认与回切） | **`/深度` `/大神` `/真神`** → **先确认** → 切对应档 → **做完回默认** |
| 4 | **自动升档** | 无 Phoenix 定义的「五档手动挡」 | **关闭自动乱切**；贵档 **手动触发**，减少 **误切 / 串 Key** |
| 5 | **模型与线路** | 全局一套为主，自管 URL / Key | **分档绑定**：日常 / 兜底 / 高档可 **不同模型、不同 Base URL、不同 `api_key_env`**（也可故意全相同，见 §5） |
| 6 | **密钥存放** | 通常在 **`.env` / `config.yaml`**，由用户习惯决定 | **Key 只进本机 `~/.hermes/.env`**；**`config.json` 只写 `api_key_env` 名**，不打进安装包 |
| 7 | **配置权威（Phoenix 侧）** | 无 `phoenix_open` | **`~/.hermes/phoenix/config.json`** 内 **`phoenix_open.model_tiers` + `providers`** |
| 8 | **Hermes 全局配置** | 仅 Hermes 项 | **追加** `phoenix.*` 路由字段、插件列表、皮肤等（安装器写入） |
| 9 | **插件** | 无 `phoenix_full` | **`phoenix_full`** 同步到 **bundled + 用户插件双路径** |
| 10 | **记忆 / 压缩 / 检查点** | 依 Hermes 自身与现有配置 | 安装器打开一批 **推荐项**（如压缩、检查点、memory 开关等），与交付文档一致 |
| 11 | **自愈 / 账本** | 无 Phoenix 模块 | Phoenix 自带 **自愈、TokenLedger（脱敏）** 等能力（见技术细则文档） |
| 12 | **省 Token 体感** | 依赖用户自己控上下文与模型 | **压缩阈值 + 检查点 + 日常不乱切贵模型** → 更易控成本 |
| 13 | **验收** | `hermes` / gateway 能跑即可 | **`doctor.py --verify`** + 对话用例（普通句不弹贵档；`/深度` 弹确认） |
| 14 | **升级关系** | 只跟 Hermes 版本 | Hermes 升级后 **`auto_fusion` + doctor**；大改再发 **新 Phoenix zip** |
| 15 | **TUI 斜杠** | 仅 Hermes 内置 `/command` | **`/深度` 等** 在 TUI 中需 **Hermes 网关层桥接**（见 **`HERMES_TUI_与不死鸟斜杠指令.md`**），否则可能被当成未知命令 |

## 3. 架构体感（适合做「上下叠图」）

**未装不死鸟（示意）**

```text
用户 → Hermes（Gateway / TUI / IM）→ 默认模型 + 单线路 Key → 回复
```

**已装不死鸟（示意）**

```text
用户 → Hermes（同上）
         → 插件 phoenix_full（拦截对话）
            → 日常：当前默认模型
            → /深度|/大神|/真神：确认 → 切档执行 → fallback/emergency → 回默认
         → Phoenix 配置 / 记忆 / 自愈 / 账本（并行存在）
```

## 4. 安装过程中：密钥「当时接」还是「事后接」？

| 策略 | 说明 | 稳定性 |
|------|------|--------|
| **当时接** | 安装器问 Base URL / 模型名 / Key 时 **直接填** | 装完 **`doctor` + gateway restart** 后，**立刻能测** 日常与高档 |
| **事后接** | 安装时 Key **回车跳过**，之后在 **`~/.hermes/.env`** 补变量，再 **`hermes gateway restart`** | **同样稳定**；`doctor --verify` 会报 **`env_key_missing_or_empty`** 直到补全 |
| **混合** | 日常先接上，高档稍后接 | **可**；高档未接 Key 时，**确认后调用可能失败**，但不会「静默乱切」 |

结论：**手动档的稳定来自「显式确认 + 绑定清晰」**；Key 早接晚接均可，**晚接就要接受高档暂时不可用**，直到 `.env` 补齐并重启网关。

## 5. 「全员单模型」 vs 「多模型切换」— 学员两种合法形态

| 形态 | 配置做法 | 产品效果 |
|------|----------|----------|
| **与你一样：单模型托底** | 日常 / 兜底 / **高档同一 Base URL + 同一模型名 + 复用 Key** | 仍有 **确认框与路由链**，但 **能力上不换脑**；适合 **先轻量上线** |
| **多模型切换** | 深度 / 大神 / 真神 绑定 **更强或不同厂商模型**，**各档 URL / `api_key_env` 分开** | **切换才有「换脑」意义**；适合 **演示与生产分级** |

两种都 **正确**；对外要说清：**不死鸟解决的是「可控升档 + 记忆/自愈/省 token 编排」**，不是强迫每人立刻买三套 API。

---

## 6. 为什么强调「手动」而不是「全自动升档」？

| 点 | 说明 |
|----|------|
| **误触发** | 自动根据关键词/意图升档，容易 **对话里「像要深度」但并非真要贵模型** |
| **Key / Provider** | 乱切易导致 **Provider 与 Key 不匹配、串 Key、网关报错** |
| **成本可见** | 手动 + **确认框** = 用户 **明确授权** 本次贵调用 |
| **可回切** | 执行完 **回默认档**，避免 **长时间挂在贵模型** |
